# 職務経歴の更新ワークフロー

職務経歴は **MD で書いて、Claude Code に依頼して JSON に変換する** 運用。

- 元原稿: [content/career/career.md](../content/career/career.md)
- サイトが読むデータ: [content/career/career.json](../content/career/career.json)
- 型定義: [content/career/types.ts](../content/career/types.ts)
- 変換スキル: [.claude/skills/career-md-to-json](../.claude/skills/career-md-to-json/SKILL.md)

## フォーマット

`---` で企業ブロックを区切り、ブロックの中に企業エントリ → 案件エントリ（複数可）を並べる。

```
**業界**          # 客先業界。社名表示として使う（客先名は伏せる）
**雇用形態**       # 任意。例: 業務委託 / SES / 正社員
**開始年月**       # YYYY-MM
**終了年月**       # YYYY-MM または「現在」
**概要**           # 2-5 行の文章で書く（箇条書きにしない）

**案件名**
**ターゲット**      # 任意。カンマ区切り。例: BtoB, BtoC
**開始年月**       # YYYY-MM
**終了年月**       # YYYY-MM または「現在」
**メンバー**       # 任意。チーム人数（例: 5名）
**開発手法**       # 任意。例: アジャイル / ウォーターフォール / Scrum
**担当**           # 箇条書きで書く
**技術**           # カンマ区切り
**概要**           # 2-5 行の文章で書く（箇条書きにしない）
```

- 「概要」（企業／案件とも）: **MD でも JSON でも 2-5 行の文章**。skill はそのままコピーするだけで要約はしない
- 「担当」: 箇条書きで書き、`string[]` として JSON に入る
- 「ターゲット」: カンマ区切り。`string[]` として JSON に入る
- 「雇用形態」「ターゲット」「メンバー」「開発手法」は任意項目。書かなければ JSON にもキーが出ない
- 1 企業に案件が 1 件だけでも構わない

> **設計原則**: skill は「コピーするだけ」。同じ MD は何度変換しても完全に同じ JSON が出る。
> 文章の言い回しを変えたいときは MD を書き換える（後で JSON 側だけ手で直してもよいが、次の変換で MD に戻ってしまう）。

## 手順

1. `content/career/career.md` を編集する
2. Claude Code に以下のように依頼する

   > `/career-md-to-json` を実行して `content/career/career.json` を更新してください。

3. Claude Code が JSON を生成 → ビルドで反映される
4. 確認して問題なければコミット

## ID の規則

`id` は識別子として URL アンカーやキー扱いする可能性があるので、

- 半角小文字英数 + ハイフン
- 業界やプロジェクトの短縮 + 開始年（例: `fintech-2024`, `saas-2022`）
- 同じ年に複数あれば `-1` `-2` で枝番

を目安とする。skill は既存 `career.json` に同じ意味のエントリがあれば既存 id を流用する。

## 注意

- 客先名は伏せる（SES のため）。業界・規模・概要レベルにとどめる
- 技術スタックは「使ったもの」を網羅でなく代表的なものに絞る
- 「概要」を文章で書くのが運用上の少しのコスト。その代わり JSON 側に揺れが出ない
