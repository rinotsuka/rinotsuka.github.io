# OpenSpec で個人サイトを駆動する

OpenSpec の `add-profile-site` という変更を切って、proposal / design / specs / tasks の
4 アーティファクトを順に作ってから実装した。

## 効いたところ

1. **デザイン判断が文章として残る**: フレームワーク選定の代替案や却下理由が後から読み返せる。
2. **要件と実装のズレが見える**: spec の SHALL を満たしているか、実装ごとに照合できる。
3. **タスクがチェックリストで進む**: 65 個に分解されたタスクが、グループ単位の進捗として見える。

## 直したいところ

- spec を書く時点で「何を機械的に検証するか」を意識すると、タスクがそのまま CI 設計になる。

OpenSpec はメモリではなくリポジトリにすべて置けるのが良い。永続メモリ非使用方針とも相性が良い。
